# Self Study Studio 产品功能手册设计

日期：2026-07-12

状态：已确认，待用户审阅书面设计

目标产物：`docs/PRODUCT_GUIDE.md`

## 1. 目标

为 Self Study Studio 建立一份长期维护的中文产品功能手册。它同时服务两个场景：

1. 对外或阶段性演示时，帮助不了解代码的人快速理解产品价值、核心流程和实际能力。
2. 项目开发过程中，准确记录功能实现状态、验证基线、已知限制和后续方向。

手册必须以当前仓库中的真实代码为准，不能把设计稿、实施计划或路线图描述成已经可用的功能。

## 2. 权威文件与入口

- `docs/PRODUCT_GUIDE.md` 是产品功能、演示方式和实现状态的权威手册。
- `README.md` 保持面向开发者的项目入口，并增加指向产品手册的固定链接。
- `personal-learning-journal-design.md` 保留为早期产品规格和产品原则来源，不继续承担实时功能状态说明。
- `docs/superpowers/specs/` 与 `docs/superpowers/plans/` 中的文件继续记录设计和实施过程；其中的能力只有进入产品代码并通过验证后，才能在产品手册中标为“已实现”。

## 3. 目标读者

正文采用非工程化表达，面向产品演示、个人回顾和新读者。附录面向开发维护，允许包含测试命令、数据存储、实现边界和对应代码模块。

读者不需要先阅读 README、源码或旧 PRD，也应能回答：

- 这个 App 解决什么问题？
- 当前可以做什么？
- 一次完整使用流程是什么？
- 如何完成一场可信的产品演示？
- 哪些能力只是设计中，尚不可演示？
- 当前版本的验证情况和已知问题是什么？

## 4. 文档结构

最终手册按以下顺序组织。

### 4.1 文档元信息

- 文档版本
- 最近核对日期
- 对应 Git commit
- 当前产品阶段
- 当前验证结果摘要

### 4.2 产品概览

- 一句话定位
- 目标用户
- 用户问题
- 核心价值
- 产品边界
- `学习项目 -> Session -> Proof -> 周复盘` 核心闭环

### 4.3 产品原则与核心概念

说明 Project、Next Step、Session、Proof、Trail 和 Review 的定义与关系，强调：

- 手机优先和低摩擦记录
- 继续学习优先于整理信息
- Proof 比完成百分比更重要
- 日常记录与阶段判断分离
- AI 不进入日常记录主流程
- 本地优先和用户数据可导出

这一节增加一张“学习轨迹闭环图”，直观表达：

```text
选择学习项目 -> 明确 Next Step -> 完成 Session -> 留下 Proof
       ^                                           |
       |------------- Review 做出决定 -------------|
```

图中同时标出 Quick Log 与 Timer 两种入口最终都会生成 Session，避免读者误以为它们是两套数据模型。

### 4.4 信息架构

说明当前三个底部入口：Today、Projects、Library，以及按条件出现的 Review 和工具栏中的 AI Review Settings。

只描述当前代码已经提供的导航结构。Calendar、Course Plan 和 Cloud Sync 作为规划能力进入状态附录，不出现在当前导航说明中。

这一节增加两张图：

1. **App 信息架构图**：展示 Today、Projects、Library 三个主入口，以及 Review、Quick Log、Timer、Proof Detail、AI Review Settings 等二级页面的进入关系。
2. **功能模块关系图**：展示界面层、业务服务、数据存储、附件存储、AI Provider 和导出服务之间的边界。

优先复用和核对仓库中已有的 `diagrams/app-user-journey.*` 与 `diagrams/app-feature-modules.*`；只有当已有图与当前代码不一致时才更新或重绘。

### 4.5 详细功能说明

每个模块使用统一模板：

- 用户目的
- 入口
- 前置条件
- 主流程
- 关键字段和操作
- 系统规则
- 空状态、失败状态和降级方式
- 与其他模块的关系
- 当前实现状态

覆盖范围：

1. 首次启动与 1–3 个学习项目创建
2. 首次 Session 门槛
3. Today Continue 卡片与 Review 提醒
4. Project 创建、编辑、状态管理和详情页
5. Quick Log
6. Timer Session
7. Proof 创建、附件类型与预览
8. Learning Trail
9. Weekly Review
10. 本地规则复盘和 OpenAI-compatible AI 复盘
11. Review 编辑、来源引用和显式应用建议
12. Library 浏览与筛选
13. JSON、附件和完整 Bundle 导出
14. SwiftData 本地存储与旧 JSON 一次性导入
15. AI Review 配置和 Keychain API Key 保存

### 4.6 端到端用户流程

至少包含：

- 第一次使用闭环
- 日常快速记录闭环
- 计时学习与 Proof 闭环
- 项目停滞与周复盘闭环
- 数据导出闭环

每个端到端流程都配一张紧邻正文的流程图，不把所有流程挤进一张总图。计划提供：

1. **首次使用流程图**：创建 1–3 个项目、选择首个项目、完成第一条 Session、进入 Today。
2. **日常记录流程图**：从 Continue 卡片选择 Quick Log 或 Timer，保存 Session，更新 Next Step，可选添加 Proof。
3. **Proof 流程图**：选择图片、录音、文件或链接，填写“这证明了什么”，保存并进入 Trail/Library。
4. **周复盘流程图**：满足提醒条件、聚合 Session 与 Proof、选择 AI 或本地规则、编辑结果、显式应用状态或 Next Step。
5. **AI 降级流程图**：未配置、请求失败或响应不可解析时，切换到本地规则复盘，并保留用户可继续操作的路径。
6. **导出流程图**：选择 JSON、附件或完整 Bundle，生成输出，并说明文件结构。

流程图使用 Mermaid 作为可维护源文件，同时输出 SVG 和 PNG 到 `diagrams/`：SVG 用于仓库 Markdown 展示，PNG 用于不支持 SVG 的演示环境。节点和分支必须使用产品语言，不暴露无助于理解的内部类名。

### 4.7 标准 Demo 指南

提供一套 5–10 分钟演示脚本，使用三个示例项目：CS336、吉他弹唱、DaVinci 调色。

Demo 必须包括：

- 演示目标和适用受众
- 演示前环境准备
- 建议预置数据
- 分步骤操作路径
- 每一步建议讲解词
- 预期屏幕结果
- 可跳过步骤和备用路径
- AI 未配置或网络不可用时的本地复盘降级演示
- 当前不可演示能力清单

演示不能依赖未验证的真机安装、CloudKit、Calendar 或 AI 课程规划。

Demo 指南采用“步骤 + 实际界面图 + 标注 + 讲解词”的形式。演示图必须来自当前可运行 App 的真实模拟器界面，不使用与实现不一致的概念稿冒充产品截图。

计划提供以下演示图：

1. **Demo 全程故事板**：用一张横向步骤图概览“Today 继续学习 -> Timer/Quick Log -> 添加 Proof -> 查看 Trail -> Weekly Review”的演示节奏。
2. **Today 起点图**：标注 Continue 卡片、Next Step、Start、Quick Log 和 Review 提醒。
3. **Quick Log 图**：标注项目默认值、学习类型、时长、记录内容和新的 Next Step。
4. **Timer 图**：标注暂停、继续、结束、舍弃和活动时长。
5. **Proof 图组**：至少包含添加入口和 Proof Detail；用局部标注说明附件类型与“这证明了什么”的必填规则。
6. **Project Trail 图**：标注 Session、Proof、Next Step、状态变化和 Review 如何串成时间线。
7. **Weekly Review 图**：标注 Facts、Patterns、Decisions、来源引用、编辑入口和显式应用建议按钮。
8. **Library 与导出图**：标注 Proof 浏览、筛选和导出入口。

截图生成规则：

- 使用 iPhone 16 Pro Simulator 或当前可用的等价 iPhone 模拟器，并在图注中记录设备和系统版本。
- 使用统一的 CS336、吉他弹唱、DaVinci 调色演示数据，保证前后截图讲述同一条故事。
- 截图保存到 `docs/assets/product-guide/`，文件名使用稳定的步骤编号，例如 `demo-01-today.png`。
- 原始截图保持真实；需要强调操作点时，另存带编号或箭头标注的演示版本，不覆盖原图。
- 标注控制在每图 3–5 个，正文逐项解释，避免图面变成密集说明书。
- 隐藏 API Key、个人文件名、真实日历信息和其他敏感数据。
- 如果某一步无法从当前构建稳定复现，则改为文字说明并标为限制，不生成伪造界面。

手册在 Demo 章节先展示全程故事板，再按演示顺序嵌入截图。每张图下方包含“操作、讲解、预期结果、失败时备用路径”四项简短说明。

### 4.8 功能状态矩阵

统一使用四种状态：

- **已实现**：产品代码中存在，并至少有自动化测试或成功构建/运行证据。
- **部分实现**：主流程存在，但有关键分支、平台验证或稳定性缺口。
- **已设计**：已有正式设计或实施计划，但产品代码尚未提供。
- **未规划**：不在当前产品范围内，或尚无已确认方案。

状态矩阵至少记录能力、状态、用户可见结果、验证证据、限制和下一步。

CloudKit/iCloud 同步、AI 课程规划和学习日历当前标为“已设计”，除非后续代码核对表明它们已经实现。

### 4.9 当前限制与已知问题

明确区分：

- 产品范围限制
- 平台与权限限制
- 尚未完成的真机验证
- 真实测试失败
- 设计中能力

截至 2026-07-12 的测试基线为 50 个测试执行、49 个通过、1 个失败。失败用例是 OpenAI-compatible Review 响应解析测试：测试响应没有提供 `sourceReferences`，而断言期望决策能够得到来源引用；当前解码逻辑会在该字段缺失时返回空字典。手册只记录该事实和用户影响，不在文档任务中顺带修改产品代码。

### 4.10 路线图与非目标

路线图按“已设计、开发中、已验证”表达，不承诺日期。明确非目标包括账号、社交、排行榜、连续打卡压力、课程市场、多人协作和全自动学习 Agent。

### 4.11 维护说明与变更记录

文档末尾保留维护规则、检查清单和按日期倒序的变更记录。

## 5. 写作与呈现规则

- 正文使用中文；保留 Today、Project、Session、Proof、Trail、Review、Next Step 等已经形成稳定含义的产品术语。
- 首次出现英文术语时给出中文解释，之后保持用词一致。
- 先讲用户价值和行为，再讲实现细节。
- 不用“支持”“完成”等模糊词掩盖部分实现状态。
- 不使用没有代码或验证证据的完成性表述。
- 演示步骤使用编号列表；功能对照与状态使用表格。
- 图与正文必须互补：图负责结构、顺序和界面定位，正文负责规则、边界和例外，不重复照抄图中的全部文字。
- 关键端到端流程至少一图，主要演示步骤至少一张实际界面图；纯字段列表和简单单步操作不强行配图。
- 所有图都提供明确标题、图注、替代文本和相对路径，保证 GitHub、Codex 和本地 Markdown 阅读时能够理解。
- 同一概念只保留一张权威图；已有流程图经过核对后直接复用，避免多个版本互相冲突。
- 流程图配色和形状同时表达含义，不依赖颜色作为唯一编码；截图标注不遮挡关键内容。

## 6. 状态判断与证据来源

判断优先级如下：

1. 当前产品源码和可运行界面
2. 当前自动化测试
3. 当前构建与模拟器运行结果
4. README 中的实现说明
5. 产品规格、设计文档和实施计划

若来源冲突，高优先级来源覆盖低优先级来源，并在“当前限制与已知问题”中说明差异。

## 7. 持续更新机制

以下变化必须同步更新产品手册：

- 新增、删除或改变用户可见功能
- 修改导航、表单字段、状态或关键业务规则
- 改变数据存储、同步、导出或迁移行为
- 改变 Demo 路径或预置数据要求
- 设计能力进入开发、实现或验证阶段
- 测试基线、构建状态或已知限制发生变化

每次更新执行以下检查：

1. 更新文档版本、核对日期和对应 commit。
2. 更新相关详细功能说明。
3. 更新功能状态矩阵。
4. 验证 Demo 路径仍与当前界面一致。
5. 更新测试基线和已知问题。
6. 在变更记录中增加一条摘要。
7. 若 README 的能力摘要发生变化，同步 README。

维护约定写入产品手册本身，并在 README 的产品文档入口旁提示：用户可见行为变化需要同步更新该手册。这样后续开发者和 Agent 都能从仓库入口发现维护要求。

## 8. 验收标准

最终产物满足以下条件才算完成：

- 不阅读源码也能理解产品定位、全部当前功能和核心流程。
- 可以严格按照文档完成一场 5–10 分钟 Demo。
- 已实现、部分实现、已设计和未规划之间没有混淆。
- 当前 50 个测试中的 1 个失败被如实记录。
- AI 课程规划、CloudKit/iCloud 同步和学习日历没有被描述为已上线。
- README 提供产品手册入口和同步维护提示。
- 文档不包含未决事项或占位标记。
- 文档包含版本、核对日期、commit、验证基线和变更记录。
- 产品概览、信息架构和六个端到端流程均有相应流程图。
- Demo 章节包含一张全程故事板和不少于六张真实模拟器演示图。
- 所有演示图与当前可运行界面一致，不把概念稿作为真实产品截图。
